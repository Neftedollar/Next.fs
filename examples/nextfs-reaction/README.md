# nextfs-reaction

Demonstrates async reactive streams (AsyncRx) inside a Next.js App Router project driven by F# and Fable.

Uses `FSharp.Control.AsyncRx` — the core engine behind [Fable.Reaction](https://github.com/dbrattli/Fable.Reaction) — wired into a Feliz `[<ReactComponent>]` via `useReducer`, `useMemo`, and `useEffect`. The `Fable.Reaction` wrapper package itself is not used because its 3.x release depends on `Fable.React 6.x` whose `ReactElement` type is incompatible with modern `Feliz`/`Fable.React.Types`.

The example shows:

- `AsyncRx.subject` bridging imperative dispatch into a reactive observable
- `AsyncRx.flatMapLatest` to cancel in-flight saves automatically
- `AsyncRx.interval` merged into the message stream for auto-increment
- `AsyncRx.toObservable` converting to `IObservable` for React effect integration
- Tag-based re-subscription to reconfigure streams on model change

## How it works

### Reactive streams vs Elmish Cmd

In Elmish, side-effects are expressed as `Cmd<Msg>` values returned from `update`:

```fsharp
// Elmish: update returns Model * Cmd<Msg>
let update msg model =
    match msg with
    | Save -> { model with Saving = true }, Cmd.OfAsync.perform save model.Count SaveDone
```

With AsyncRx, `update` is **pure** — it returns only the new model. All side-effects live in a `stream` function that transforms the raw message observable using reactive operators:

```fsharp
// AsyncRx: update returns Model only
let update model msg =
    match msg with
    | Save _ -> model
    | SaveStarted -> { model with SaveStatus = Saving }
    | SaveComplete n -> { model with SaveStatus = Saved n }
```

```fsharp
// Side-effects are reactive stream transformations
let buildStream model (msgs: IAsyncObservable<Msg>) =
    let saves =
        msgs
        |> AsyncRx.choose (function Save count -> Some count | _ -> None)
        |> AsyncRx.flatMapLatest (fun count ->
            AsyncRx.ofAsync (simulateSave count)
            |> AsyncRx.map SaveComplete
            |> AsyncRx.startWith [ SaveStarted ])
    ...
```

### Key reactive concepts

**Stream merging** — user clicks and a timer interval are merged into one message stream:

```fsharp
let ticks = AsyncRx.interval 0 1000 |> AsyncRx.map (fun _ -> AutoTick)
combined |> AsyncRx.merge ticks
```

**Tag-based re-subscription** — the component tracks a string tag. When it changes (`"manual"` ↔ `"auto"`), the previous subscription is disposed and a new one is created. This is how the timer starts and stops:

```fsharp
if model.AutoRun then
    combined |> AsyncRx.merge ticks, "auto"
else
    combined, "manual"
```

**Automatic cancellation** — `flatMapLatest` cancels an in-flight save when a new one arrives. In Elmish you would need manual cancellation tracking.

### React integration

`Counter.fs` is a `[<ReactComponent>]` that uses Feliz hooks:

1. `React.useReducer` — manages model state (pure update, no Cmd)
2. `AsyncRx.subject` (via `useMemo`) — bridges imperative dispatch into a reactive observable
3. `AsyncRx.toObservable` (via `useMemo`) — converts async observable to sync `IObservable` for React effect compatibility
4. `React.useEffect` — subscribes/re-subscribes when the tag changes
5. `React.useEffectOnce` — cleans up the final subscription on unmount

`Counter` is a plain F# module — it has no `[<NextFs.NextFsEntry>]` attribute. It is imported by `App.Page`, which carries the `Directive="use client"` annotation. Next.js treats everything in the import tree of a `'use client'` entry as client-side code, so `Counter` inherits the boundary automatically.

## Running locally

From the repository root:

```bash
dotnet tool restore
dotnet restore examples/nextfs-reaction/src/NextFs.Reaction.Example.fsproj
```

From this directory:

```bash
npm install
npm run sync:app   # build:fable → scan → gen:wrappers
npm run dev
```

For iterative work:

```bash
npm run watch:fable   # re-emits + re-scans + re-generates on every F# save
npm run dev
```

## Adding a real server action

The simulated save in `Counter.fs` calls `Async.Sleep 600` and returns the count unchanged. To replace it with a real Next.js server action:

**1. Create a server actions module:**

```fsharp
[<NextFs.NextFsEntry("app/actions.js", Directive="use server", Named="saveCount")>]
module App.Actions

open NextFs

let saveCount (count: int) : int =
    Directive.useServer()
    count
```

**2. Import the action in `Counter.fs` from the generated wrapper:**

```fsharp
open Fable.Core

[<Import("saveCount", "../../app/actions.js")>]
let private saveCount: int -> JS.Promise<int> = jsNative
```

The import path `"../../app/actions.js"` is relative to the Fable output location
`.fable/App/Counter.js`. It resolves to the generated server-action wrapper.

**3. Replace `simulateSave`:**

```fsharp
let private doSave (count: int) : Async<int> =
    async { return! Async.AwaitPromise(saveCount count) }
```

**4. Add `App/Actions.fs` to the fsproj** before `App/Counter.fs`.

## Limitations

### Server components cannot use reactive streams

React Server Components run on the server and cannot hold state or subscribe
to observables. Use the client-component pattern shown here and pass initial
data from a server parent as props if needed.

### Module boundary for server action imports

When a client module imports a Next.js server action, the import must
reference the generated `app/actions.js` wrapper (which carries the file-level
`'use server'` directive), not the raw `.fable/App/Actions.js` Fable output.
Use a relative path from the Fable output location as shown above.

### Fable.Reaction 3.x compatibility

`Fable.Reaction` 3.5.0 depends on `Fable.React >= 8.0` (actually resolves to
6.2.0), whose `ReactElement` type differs from the one in
`Fable.React.Types 18.x` used by modern `Feliz`. This example therefore uses
`FSharp.Control.AsyncRx` directly and wires it into React via Feliz hooks.
If a future `Fable.Reaction` release aligns with `Fable.React.Types`, you
could use `Reaction.streamComponent` instead of the manual hook wiring shown
here.
