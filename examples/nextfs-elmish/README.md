# nextfs-elmish

Demonstrates the Elmish MVU pattern inside a Next.js App Router project driven by F# and Fable.

The example shows:

- `React.useElmish` from `Feliz.UseElmish` in a `'use client'` component
- Async `Cmd.OfAsync.perform` for side-effecting messages (simulated save)
- How to structure an Elmish component so it sits below a Next.js entry boundary
- Where to plug in real Next.js server actions

## Tree

```text
examples/nextfs-elmish/
├── app/
│   ├── layout.js           # generated
│   └── page.js             # generated (use client)
├── next.config.mjs
├── nextfs.entries.json     # generated by npm run scan
├── package.json
└── src/
    ├── NextFs.Elmish.Example.fsproj
    └── App/
        ├── Counter.fs      # Elmish MVU component (no entry annotation)
        ├── Layout.fs       # root layout + metadata
        └── Page.fs         # client entry, mounts Counter
```

## How it works

`Counter.fs` contains the full MVU loop using `Feliz.UseElmish`:

```fsharp
[<ReactComponent>]
let Counter () =
    let model, dispatch = React.useElmish(init, update)
    ...
```

`Counter` is a plain F# module — it has no `[<NextFs.NextFsEntry>]` attribute. It is imported by `App.Page`, which carries the `Directive="use client"` annotation. Next.js treats everything in the import tree of a `'use client'` entry as client-side code, so `Counter` inherits the boundary automatically.

`Page.fs` is the actual entry point:

```fsharp
[<NextFs.NextFsEntry("app/page.js", Directive="use client", Default="Page")>]
module App.Page

open App.Counter

let Page () =
    Html.main [ ... Counter() ... ]
```

## Running locally

From the repository root:

```bash
dotnet tool restore
dotnet restore examples/nextfs-elmish/src/NextFs.Elmish.Example.fsproj
```

From this directory:

```bash
npm install
npm run sync:app   # build:fable → scan → gen:wrappers
npm run dev
```

For iterative work:

```bash
npm run watch:fable   # re-emits + re-scans + re-generates on every F# save
npm run dev
```

## Adding a real server action

The simulated save in `Counter.fs` calls `Async.Sleep 600` and returns the count unchanged. To replace it with a real Next.js server action:

**1. Create a server actions module:**

```fsharp
[<NextFs.NextFsEntry("app/actions.js", Directive="use server", Named="saveCount")>]
module App.Actions

open NextFs

// Persist count and return the saved value.
// In a real project, call your database here.
let saveCount (count: int) : int =
    Directive.useServer()
    count
```

**2. Import the action in `Counter.fs` from the generated wrapper:**

```fsharp
open Fable.Core

[<Import("saveCount", "../../app/actions.js")>]
let private saveCount: int -> JS.Promise<int> = jsNative
```

The import path `"../../app/actions.js"` is relative to the Fable output location
`.fable/App/Counter.js`. It resolves to the generated server-action wrapper.

**3. Replace `simulateSave`:**

```fsharp
let private doSave (count: int) : Async<int> =
    async { return! Async.AwaitPromise(saveCount count) }
```

**4. Add `App/Actions.fs` to the fsproj** before `App/Counter.fs`.

## Limitations

### `Program.withReact*` does not work

The classic Elmish SPA entry point mounts directly to a DOM node:

```fsharp
Program.mkProgram init update view
|> Program.withReactSynchronous "app-root"  // ← calls ReactDOM.createRoot
|> Program.run
```

Next.js owns the React root. There is no `"app-root"` div in your HTML to target,
and calling `ReactDOM.createRoot` from an App Router module will throw at runtime.

Use `React.useElmish` from `Feliz.UseElmish` instead. It wraps the Elmish loop
in a `useReducer` hook that React manages as part of the component tree.

### Elmish cannot drive App Router navigation

Routing in Next.js App Router is file-system based. Elmish cannot intercept or
replace it. Use `NavigationClient.useRouter()` in a message handler if you need
to navigate programmatically:

```fsharp
open NextFs

// In a [<ReactComponent>]:
let router = NavigationClient.useRouter()

// In update:
| NavigateToDashboard ->
    model,
    Cmd.ofEffect (fun _ -> router.push "/dashboard")
```

### Server components cannot use Elmish

React Server Components run on the server and cannot hold state or call React
hooks. Elmish is pure client-side. Use the client-component pattern shown here
and pass initial data from a server parent as props if needed:

```fsharp
// Server component (no directive):
let Page () =
    let initialCount = fetchCountFromDb()  // async server call
    Html.div [ CounterWithInit initialCount ]

// Client component (use client):
[<ReactComponent>]
let CounterWithInit (initial: int) =
    let model, dispatch = React.useElmish((fun () -> init initial), update)
    ...
```

### Module boundary for server action imports

When a client Elmish module imports a Next.js server action, the import must
reference the generated `app/actions.js` wrapper (which carries the file-level
`'use server'` directive), not the raw `.fable/App/Actions.js` Fable output.
Use a relative path from the Fable output location as shown in the "Adding a
real server action" section above.
